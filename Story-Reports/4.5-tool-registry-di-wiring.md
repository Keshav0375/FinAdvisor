# 4.5 — Tool Registry + DI Wiring

## Problem
The agent orchestrator needs a single entry point to discover available tools, generate Anthropic-compatible schemas, and dispatch tool calls with proper dependency injection — without the orchestrator knowing each tool's specific dependencies.

## Implementation
- `backend/src/agent/tools/__init__.py` — `ToolRegistry` class that accepts `AsyncSession` + `VoyageEmbeddings` at init, creates a `VectorStore` internally, and exposes: `to_anthropic_schema()` for Claude API tool list, `tool_names` property, and `execute(name, input, user)` async dispatcher that validates input via Pydantic models and routes to the correct tool function
- `backend/tests/test_tool_registry.py` — 8 unit tests: schema returns all 4 tools, schemas have required fields (name/description/input_schema), tool_names property, escalate dispatch returns correct JSON, unknown tool raises ValueError, search_kb dispatch with mocked embeddings+vector_store, output is JSON string, invalid input raises ValidationError

## Decisions
- Adapted the ARCHITECTURE.md class-per-tool pattern to dispatch directly to existing async functions — avoids wrapping each tool in a class when the functions already have the right signatures.
- Registry owns VectorStore creation from the injected session — keeps the orchestrator from needing to know about VectorStore.
- `execute()` returns `str` (JSON) rather than a Pydantic model — matches what the Anthropic API expects in tool_result content blocks.
- Used `type: ignore[arg-type]` for the narrowing casts from BaseModel to specific input types — mypy can't infer the runtime dispatch, but the model_validate call guarantees the correct type.
