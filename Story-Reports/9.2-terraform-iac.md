# 9.2 — Terraform IaC

## Problem
The FinAdvisor architecture needed infrastructure-as-code to provision GCP resources reproducibly across environments (dev, staging, prod). Without Terraform, deployment would be manual and error-prone.

## Implementation
- **`terraform/main.tf`** — Google provider config, GCS backend for terraform state, `required_version >= 1.5`
- **`terraform/variables.tf`** — 14 input variables: project_id, region, environment, image tags, DB tiers, resource limits
- **`terraform/artifact_registry.tf`** — Docker repo for container images, registry URL local
- **`terraform/cloud_sql.tf`** — 2 Cloud SQL Postgres 16 instances:
  - App DB with `cloudsql.enable_pgvector = on`, env-aware backup config, deletion protection in prod
  - LangFuse DB (separate instance for isolation)
  - Random passwords, DB URL composed and stored in Secret Manager
- **`terraform/cloud_run.tf`** — 5 Cloud Run v2 services:
  - Backend (with Cloud SQL proxy volume, Secret Manager env vars for all API keys, GCP DLP mode)
  - Frontend (NEXT_PUBLIC_API_URL pointing to backend service URL)
  - LiteLLM (API keys from Secret Manager)
  - Kong (DB-less mode, declarative config)
  - LangFuse (Cloud SQL proxy volume, auto-composed DB URL)
- **`terraform/cloud_storage.tf`** — 2 GCS buckets: corpus data (versioned, lifecycle rules), eval results
- **`terraform/secrets.tf`** — 9 Secret Manager secrets for all API keys, DB URL, JWT secret, LangFuse keys
- **`terraform/iam.tf`** — 2 service accounts (backend, frontend), IAM bindings for Secret Manager access, Cloud SQL client, DLP user, Storage viewer, public Run invoker
- **`terraform/outputs.tf`** — 10 outputs: all service URLs, DB connection names, registry URL, bucket names

## Decisions
- Separated app DB and LangFuse DB into distinct Cloud SQL instances — LangFuse can be torn down without affecting the core application.
- Environment-aware configuration: prod gets regional HA, backups, deletion protection, min 1 instance; dev/staging gets zonal, no backup, no protection, scale-to-zero.
- Cloud SQL proxy via volume mounts rather than public IP — follows GCP best practices for security.
- All secrets created as Secret Manager resources without initial versions — secret values are populated manually or via CI, never in Terraform state.
- Random passwords generated by Terraform's `random_password` provider for DB users.
