# 2.2 — PII Redaction Module

## Problem
The ingest pipeline must redact PII before embedding documents into the vector store (CLAUDE.md non-negotiable rule #2). Need both a production-grade GCP DLP integration and a regex fallback for local dev without GCP credentials.

## Implementation
- `backend/src/pii/__init__.py` — Factory function `create_redactor()` that selects GCP DLP or regex based on `Settings.pii_mode`
- `backend/src/pii/redactor.py` — `PIIRedactor` class wrapping GCP DLP v2:
  - Detects 8 info types: PERSON_NAME, PHONE_NUMBER, EMAIL_ADDRESS, US_SSN, CA_SIN, FINANCIAL_ACCOUNT_NUMBER, CREDIT_CARD_NUMBER, STREET_ADDRESS
  - Uses `deidentify_content` with `replace_with_info_type_config`
  - Returns `RedactionResult` (redacted_text + findings list)
- `backend/src/pii/fallback.py` — `RegexRedactor` class for local dev:
  - Patterns: SSN (`\d{3}-\d{2}-\d{4}`), phone (with parens/dashes/dots), email, account numbers
  - Same `RedactionResult` interface as DLP version
- `backend/tests/test_pii.py` — 10 tests:
  - 7 regex tests (SSN, phone variants, email, account, multi-PII, no-PII)
  - 1 DLP mock test (verifies deidentify_content call and response parsing)
  - 2 factory tests (regex default, DLP when configured)

## Decisions
- Both redactors share the same return type (`RedactionResult` dataclass with `findings` list) so the ingest pipeline doesn't need to know which backend is active.
- Used `dataclass` instead of Pydantic for internal data structures — these don't cross API boundaries.
- Regex fallback catches the same PII types embedded in the synthetic corpus (phone, email in privacy notices) — verified this covers the test corpus.
- Phone pattern handles `(555) 123-4567` and `555-123-4567` and `555.123.4567` variants.
- DLP redactor is synchronous (google-cloud-dlp SDK is sync) — will be called via `asyncio.to_thread()` in the ingest pipeline.
