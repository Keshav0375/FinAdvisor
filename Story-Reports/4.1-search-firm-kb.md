# 4.1 — Tool: search_firm_kb

## Problem
The agent needs a tool to search the firm's knowledge base via vector similarity, with results automatically filtered by the advisor's tier and jurisdiction through PostgreSQL RLS.

## Implementation
- `backend/src/retrieval/vector_store.py` — VectorStore class with `similarity_search(query_embedding, top_k)` method using pgvector cosine distance (`1 - (embedding <=> query)`), returns dicts with chunk_id, content, source_title, regulatory_ref, last_reviewed_at, similarity_score
- `backend/src/agent/tools/search_kb.py` — Pydantic models (SearchKBInput, ChunkResult, SearchKBOutput) + `search_firm_kb()` async function that embeds query via VoyageEmbeddings → VectorStore similarity search → typed output
- `backend/src/agent/__init__.py` — Agent package init
- `backend/src/agent/tools/__init__.py` — Tools package init
- `backend/tests/test_search_kb.py` — 4 tests: mocked deps, empty results, integration with RLS, VectorStore RLS count verification

## Decisions
- VectorStore takes an AsyncSession (injected), not its own engine — the session must already have RLS context set via `set_rls_context()`.
- `search_firm_kb()` takes VectorStore and VoyageEmbeddings as explicit parameters (DI) rather than constructing them internally — makes testing straightforward and follows CLAUDE.md's "no singletons, no global state" rule.
- Used raw SQL with `text()` for the vector similarity query rather than ORM — the `<=>` operator and `cast(:vec as vector)` are cleaner in raw SQL and avoid pgvector/SQLAlchemy typing complexity.
- SEARCH_KB_TOOL dict follows the Anthropic tool schema format from ARCHITECTURE.md — ready for tool registry integration.
