# 4.6 — Agent Orchestrator (ReAct Loop)

## Problem
The system needs a core agent loop that takes a user query, calls Claude with tools, executes tool calls, feeds results back, and loops until Claude produces a final text response — all while streaming events to the caller and recording traces for observability.

## Implementation
- `backend/src/agent/types.py` — Pydantic models for `StreamEvent` (type: text|tool_call|tool_result|citation|error), `ToolCallEvent` (tool_name, tool_input, tool_use_id), `CitationEvent` (index, source_title, regulatory_ref, last_reviewed_at)
- `backend/src/agent/system_prompt.py` — Versioned system prompt (v1.0.0) from ARCHITECTURE.md Section 6 with `get_system_prompt()` helper. Covers all 6 critical rules: citation enforcement, unsourced claim refusal, license escalation, no fabricated refs, stale doc warnings
- `backend/src/agent/orchestrator.py` — `FinAdvisorAgent` class: takes `AsyncAnthropic` client + `ToolRegistry` + optional LangFuse. `run()` async generator implements ReAct loop: query → Claude → tool_use → execute → loop → text. Max 5 iterations with error event on exhaustion. Tool errors caught and returned as JSON error payloads. LangFuse creates trace per query, generation per iteration, span per tool call
- `backend/tests/test_agent.py` — 8 tests with fully mocked LLM: single-turn text, tool_use→text loop, max iterations error, tool execution error handling, LangFuse trace integration, multiple tool calls in one response, system prompt content verification, version check

## Decisions
- Used `AsyncAnthropic` (not sync `Anthropic`) since the entire backend is async — matches CLAUDE.md requirement
- LangFuse is optional (`None` = no-op) — allows running without LangFuse in dev/test while still having full tracing in production
- `cast(Any, ...)` for messages/tools params — the Anthropic SDK uses complex union types that mypy can't coerce from `list[dict]`, but the runtime accepts them fine
- Tool errors return JSON `{"error": "..."}` rather than raising — lets Claude decide how to handle the failure (retry, apologize, try different tool)
- Yields `tool_call` and `tool_result` events alongside `text` — the SSE endpoint (Phase 6) can forward these to the frontend for real-time tool call visibility
