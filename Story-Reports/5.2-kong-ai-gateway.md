# 5.2 — Kong AI Gateway Configuration

## Problem
The LLM proxy needs a gateway layer for per-user rate limiting, request size protection, and audit logging before requests reach LiteLLM.

## Implementation
- `kong/kong.yml` — Declarative config (format v3.0): service `llm-service` routing to `http://litellm:4000` on path `/v1`. Three plugins: rate-limiting (60 req/min, 500 req/hr per X-User-Id header), request-size-limiting (1MB payload cap), http-log (POSTs request/response audit to backend at `/api/audit/kong`)
- `kong/Dockerfile` — Based on `kong:3.9`, copies kong.yml, sets DB-less mode via env vars, exposes proxy and admin ports
- `docker-compose.yml` — Already had correct wiring from task 0.3: `build: ./kong`, `depends_on: litellm`, port mapping `8001:8000`

## Decisions
- Used Kong 3.9 (latest stable) in DB-less declarative mode — no database needed, config is baked into the container image.
- Rate limiting uses `header` identifier on `X-User-Id` — matches our auth pattern where every request carries this header. This means per-advisor rate limits, not per-IP.
- 1MB payload limit prevents prompt injection via massive payloads while allowing normal tool-heavy conversations (~100KB typical).
- Audit log endpoint (`/api/audit/kong`) doesn't exist yet — it will be created in Phase 6 when the backend API is built out.
