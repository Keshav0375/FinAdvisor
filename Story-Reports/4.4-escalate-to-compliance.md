# 4.4 — Tool: escalate_to_compliance

## Problem
The agent needs a compliance escalation tool to flag queries that involve products outside the advisor's licenses, regulatory conflicts, or ambiguous suitability determinations requiring human review.

## Implementation
- `backend/src/agent/tools/escalate.py` — Pydantic models (EscalateInput, EscalationRecord) + `escalate_to_compliance()` async function that generates a structured escalation record and logs a `compliance_escalation` warning via structlog
- `backend/tests/test_escalate.py` — 6 unit tests: structured output fields, escalation ID format (ESC-{advisor}-{timestamp}), ISO timestamp, structlog warning emission (mock verified), Anthropic tool schema, multi-user differentiation

## Decisions
- No database persistence — MVP logs escalations via structlog. A real system would write to an escalations table and notify compliance officers via webhook/email.
- Escalation ID uses `ESC-{user.sub}-{unix_timestamp}` format — simple, sortable, and traceable to both advisor and time.
- No DB session parameter — unlike other tools, this one is pure compute + logging, no database queries needed.
- Tool schema uses `advisor_licenses` as input (not from UserClaims) — the agent explicitly passes what licenses it detected as relevant, which creates a clear audit trail of the agent's reasoning.
