# 5.1 — LiteLLM Proxy Configuration

## Problem
The agent orchestrator needs a unified LLM gateway that routes requests to Claude as the primary model, with automatic fallback to GPT-4o and Gemini when Claude is unavailable, plus circuit breaker protection against cascading failures.

## Implementation
- `litellm/config.yaml` — Model routing config with 3 entries all behind the `claude-sonnet-4-20250514` model name: Claude primary (Anthropic SDK), GPT-4o fallback (OpenAI), Gemini 2.5 Flash fallback (Google). Router settings: simple-shuffle strategy, 2 retries, 5s retry delay, fallback chain claude-primary → gpt4o-fallback → gemini-fallback, circuit breaker (3 allowed fails, 300s cooldown). LiteLLM settings: drop_params=true for cross-provider compat, set_verbose=false. Master key from env var.
- `litellm/Dockerfile` — Based on `ghcr.io/berriai/litellm:main-latest`, copies config.yaml, exposes port 4000.

## Decisions
- Used `ghcr.io/berriai/litellm:main-latest` base image — official LiteLLM image, maintained by BerriAI, includes all provider dependencies.
- All API keys via `os.environ/` syntax — LiteLLM's native env var resolution, no keys in config files.
- `drop_params: true` — critical for cross-provider fallback. Claude-specific params (like `tools`) that aren't supported by GPT-4o/Gemini get silently dropped instead of causing errors.
- All 3 models use the same `model_name` — this is how LiteLLM fallback routing works: the backend requests `claude-sonnet-4-20250514` and LiteLLM tries providers in order.

## Verification
- YAML validated programmatically: 3 models, correct fallback chain, circuit breaker (3 fails / 300s)
- Docker verification deferred (Docker Desktop not available on this machine)
- Quality gate: ruff, mypy, pytest all pass (62 passed, 2 skipped)
