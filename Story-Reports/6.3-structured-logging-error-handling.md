# 6.3 ‚Äî Structured Logging + Error Handling

## Problem
The codebase used structlog throughout but never configured it (no JSON output, no timestamps, no request context binding). There was also no custom exception hierarchy or global error handling ‚Äî unhandled errors returned raw stack traces.

## Implementation
- **`backend/src/observability/__init__.py`** ‚Äî Package init
- **`backend/src/observability/logging.py`** ‚Äî `configure_logging()` with JSON/console modes, ISO timestamps, contextvars merging; `RequestContextMiddleware` binds request_id + user_id per request; `bind_request_context()` utility
- **`backend/src/errors.py`** ‚Äî Exception hierarchy: FinAdvisorError ‚Üí AuthorizationError, RetrievalError, ToolExecutionError, EvalThresholdError
- **`backend/src/main.py`** ‚Äî Wired `configure_logging()` in lifespan, added `RequestContextMiddleware`, registered `finadvisor_exception_handler` (maps error types to HTTP status codes) and unhandled exception catch in `request_logging_middleware`
- **`backend/tests/test_logging.py`** ‚Äî 11 tests covering JSON output format, contextvars binding, error hierarchy, 4 HTTP status code mappings, request ID middleware

## Decisions
- Caught unhandled exceptions in `request_logging_middleware` instead of using `add_exception_handler(Exception, ...)` because Starlette's `BaseHTTPMiddleware` wraps exceptions in ExceptionGroups that bypass registered handlers.
- Error status mapping: AuthorizationError‚Üí403, RetrievalError‚Üí502 (upstream dependency), ToolExecutionError‚Üí500. Unhandled errors return generic 500 without leaking internals.
- X-Request-Id header: echoed if provided by client, auto-generated UUID if not.

## Verification
- Quality gate passed: ruff (0 errors), mypy (0 errors), pytest (85 passed, 2 skipped)
- JSON log output verified: `{"key": "value", "event": "test_event", "level": "info", "timestamp": "..."}`
- All exception types produce correct HTTP status codes and structured error responses
