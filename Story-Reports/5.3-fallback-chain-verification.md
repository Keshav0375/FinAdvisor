# 5.3 — Fallback Chain Verification

## Problem
The LLM routing layer (LiteLLM + Kong) needs verification that fallback behavior works correctly — the agent should function normally regardless of which provider serves the request, and errors should propagate clearly when all providers fail.

## Implementation
- `backend/tests/test_fallback_chain.py` — 5 unit tests verifying agent behavior under various LLM provider failure scenarios: primary success, fallback transparency (LiteLLM routing is invisible to the agent), all-providers-fail error propagation (APIConnectionError), rate-limit error propagation (429), circuit-breaker simulation (3 consecutive APIStatusError 529s)
- `backend/scripts/test_fallback_live.py` — Standalone async script for Docker-based manual verification: checks LiteLLM health endpoint, sends a test request through Kong → LiteLLM → Claude, verifies Kong rate-limit headers are present in response

## Decisions
- Tests mock at the Anthropic client level, not at HTTP level — because LiteLLM handles retries/fallback transparently before the response reaches the agent. The agent never sees "I'm switching providers."
- Errors from the mock raise Anthropic SDK exception types (APIStatusError, APIConnectionError, RateLimitError) — these are what the agent would actually see when LiteLLM exhausts all fallbacks.
- Live script uses httpx instead of the Anthropic SDK — it tests the raw HTTP flow through Kong → LiteLLM, which is what validates the infrastructure wiring (Docker verification deferred).
- No retry logic added to the orchestrator — LiteLLM handles retries. Adding agent-level retries would cause double-retry loops.

## Verification
- `pytest tests/test_fallback_chain.py -v` — 5 tests pass
- Full suite: 67 passed, 2 skipped
- Quality gate: ruff, mypy, pytest all clean
- Live Docker verification deferred (Docker Desktop not available)
