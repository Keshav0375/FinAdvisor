# 3.1 — Mock User Profiles + JWT Models

## Problem
The agent and API need to identify the current user to set RLS context (tier + jurisdictions) on every database session. Need a mock auth system that's trivial to swap for real JWT verification later.

## Implementation
- `backend/src/auth/mock_users.py` — `MOCK_USERS` dict with 4 personas from ARCHITECTURE.md:
  - Sarah Chen (senior/tier-3/US), Alex Kim (associate/tier-1/EU), James Wright (private_wealth/tier-4/UK), Priya Sharma (advisor/tier-2/US+EU)
- `backend/src/auth/jwt.py` — `get_current_user` FastAPI dependency:
  - Reads `X-User-Id` header
  - Looks up in MOCK_USERS, returns `UserClaims`
  - Returns 401 for unknown/missing user
- `backend/src/auth/models.py` — `UserClaims` (already existed from task 1.3)
- `data/seed/mock_users.json` — Same 4 users as JSON for frontend user picker
- `backend/tests/test_auth.py` — 3 tests via httpx AsyncClient against a test FastAPI app
- `backend/ruff.toml` — Added `B008` to ignore list (FastAPI `Depends()` is a valid default)

## Decisions
- Header-based auth (`X-User-Id`) rather than JWT tokens — simplest possible MVP auth that still exercises the full RLS flow. Swapping to real JWT later only changes `jwt.py`, not the rest of the system.
- `UserClaims` is a Pydantic model (not dataclass) — allows easy serialization for API responses and validation at the boundary.
- `data/seed/mock_users.json` gives the frontend a way to populate a user-picker dropdown without hardcoding.

## Verification
- `mypy src/auth/` — passes (0 issues in 3 files)
- `pytest tests/test_auth.py -v` — 3 tests pass:
  - valid user → 200 + correct claims
  - unknown user → 401
  - missing header → 401
